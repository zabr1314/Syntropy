# Syntropy (太和) - 技术架构与开发白皮书

> **文档版本**: 2.0 (Stable)
> **适用范围**: Core Architecture & Agent Runtime

---

## 1. 项目定位与核心愿景

**Syntropy (太和)** 是一套基于 **"古代朝廷"** 视觉隐喻的 **可视化多智能体操作系统 (Visualized Multi-Agent Operating System)**。

我们的核心目标是解决当前 Multi-Agent 系统面临的三大痛点：
1.  **黑盒化 (Black Box)**: 用户无法直观理解 Agent 的思考过程与决策依据。
2.  **失控风险 (Uncontrollable)**: 缺乏有效的人机协同机制来拦截高风险操作。
3.  **记忆遗忘 (Amnesia)**: 传统 RAG 在长对话中容易丢失上下文，导致 Agent 变“笨”。

Syntropy 通过将 Agent 具象化为 2D 像素小人，将思考过程可视化为状态机流转，将权限管理游戏化为“御批”机制，提供了一套完整的解决方案。

## 2. 系统技术栈 (Tech Stack)

### 2.1 前端 UI 层 (文官系统)
- **Framework**: React 18 + TypeScript + Vite
- **State Management**: Zustand (单一数据源，连接 UI 与 Game)
- **Styling**: Tailwind CSS (原子化样式)
- **UI Components**: Framer Motion (动效), Lucide React (图标)

### 2.2 前端渲染层 (武将系统)
- **Engine**: Phaser 3 (WebGL/Canvas)
- **Pathfinding**: Easystar.js (A* 寻路算法)
- **Responsibility**: 负责 2D 沙盘渲染、角色动画播放、碰撞检测与寻路逻辑。不处理业务状态，只负责“表现”。

### 2.3 后端驱动层 (Agent Runtime)
- **Runtime**: Node.js (Express)
- **Communication**: Socket.io (双向实时通信)
- **Database**: Better-SQLite3 (存储记忆与向量索引)
- **Vector Search**: Local Embedding / OpenAI Embedding + Cosine Similarity

## 3. 核心架构设计

### 3.1 混合记忆系统 (Hybrid Memory Engine with RRF)
我们摒弃了单一的向量检索，采用 **Reciprocal Rank Fusion (RRF)** 算法融合多种检索策略：

1.  **全文检索 (FTS5)**: 利用 SQLite 的 FTS5 模块进行关键词匹配。
    *   *优势*: 精准定位专有名词（如“昨天的税收”、“兵部尚书”）。
2.  **向量检索 (Vector Search)**: 计算 Query Embedding 与 Memory Chunk Embedding 的余弦相似度。
    *   *优势*: 理解语义模糊的指令（如“最近有什么异常吗？”）。
3.  **记忆压缩 (Memory Compression)**: 
    *   *机制*: 当 Agent 进入 `SLEEPING` 状态时，系统自动调用 LLM 对当日未处理的对话进行摘要，生成 `daily_summary` 并持久化。这有效解决了长下文 Token 溢出问题。

### 3.2 可视化智能体运行时 (Visualized Runtime)
前端采用 **React-Phaser Bridge** 模式：
- **Single Source of Truth**: 所有 Agent 的状态（位置、动作、思考内容）均存储在 Zustand Store 中。
- **Reactive Rendering**: React 组件订阅 Store 更新 UI 面板；Phaser 场景在 `update()` 循环中读取 Store 同步小人动画。
- **ACP 协议**: 后端通过 Socket.io 推送 `agent_update` 事件，前端收到后更新 Store，实现毫秒级同步。

### 3.3 内核级状态机 (Kernel-Level State Machine)
后端 `Agent` 类维护了一个标准化的有限状态机 (FSM)：
- `INITIALIZING`: 加载配置与记忆。
- `IDLE`: 空闲等待指令。
- `THINKING`: 调用 LLM 生成思维链 (CoT)。
- `ACTING`: 执行工具调用 (Tool Call)。
- `WAITING_FOR_HUMAN`: 触发风控拦截，等待御批。
- `SLEEPING`: 休眠并执行记忆压缩。
- `ERROR`: 发生异常。

### 3.4 人机协同“御批”协议 (Human-in-the-loop)
- **风险分级**: 每个 Tool 定义了 `riskLevel` (low/medium/high)。
- **拦截机制**: 内核在执行 Tool 前检查风险等级。若为 High，则挂起执行，向前端发送 `approval_request`。
- **御批流程**: 前端弹出“御批”窗口，用户点击“准奏”或“驳回”后，Socket 发回指令，内核恢复 (`resumeFromApproval`) 执行流。

## 4. 数据协议 (Data Protocol)

### 4.1 Agent 状态更新 (Server -> Client)
```json
{
  "type": "agent_update",
  "data": {
    "id": "minister",
    "status": "thinking",
    "message": "正在查阅户部档案...",
    "targetPosition": { "x": 300, "y": 200 } // 可选，用于移动
  }
}
```

### 4.2 御批请求 (Server -> Client)
```json
{
  "type": "approval_request",
  "data": {
    "agentId": "official_revenue",
    "toolCallId": "call_12345",
    "functionName": "transfer_funds",
    "args": { "amount": 10000, "to": "war_department" },
    "riskLevel": "high"
  }
}
```

## 5. 开发指南

### 5.1 添加新技能
1. 在 `server/skills/` 目录下创建 `.js` 文件。
2. 导出一个包含 `definition` (JSON Schema) 和 `handler` (函数逻辑) 的对象。
3. 在 `server/config/officials.json` 中将新技能名称添加到对应 Agent 的 `skills` 列表中。

### 5.2 调试 RAG
1. 打开前端“内务府”面板。
2. 使用 **“记忆透视镜 (Memory Debugger)”**。
3. 选择 Agent 并输入 Query，系统将展示 FTS 排名与 Vector 排名的对比结果。

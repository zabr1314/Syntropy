# OpenClaw 会话管理机制说明

OpenClaw 的会话 (Session) 管理机制是支撑其长短期记忆、多渠道路由和状态持久化的基石。本文档详细解析会话的生命周期、存储结构及关键实现。

## 1. 会话标识符 (Session Key)

Session Key 是 OpenClaw 中定位一个会话的唯一字符串，它决定了消息被路由到哪里。

### 1.1 格式规范
标准的 Session Key 格式如下：
`agent:<AgentID>:<ScopeIdentifier>`

- **AgentID**: 归属的 Agent（如 `minister`, `historian`）。
- **ScopeIdentifier**: 会话的范围标识。
  - **用户私聊**: 通常是用户的 ID 或电话号码（如 `agent:minister:+1234567890`）。
  - **群组/频道**: 包含群组标识（如 `agent:minister:group:123`）。
  - **子代理**: 包含 UUID（如 `agent:historian:subagent:550e8400-e29b...`）。
  - **全局**: 特殊的 `global` 关键字，或 `main`。

### 1.2 解析逻辑 (`session-key.ts`)
系统通过 `deriveSessionKey` 函数根据入站消息上下文 (`MsgContext`) 自动推导 Session Key。如果消息中显式携带了 `SessionKey` 字段，则优先使用。

## 2. 会话存储 (Session Store)

OpenClaw 不依赖大型数据库，而是采用基于文件系统的高效 JSON 存储方案。

### 2.1 存储文件 (`store.json`)
每个 Agent（或全局）都有一个 `store.json` 文件，存储了该 Agent 下所有活跃会话的元数据。

**Entry 结构示例:**
```json
{
  "agent:minister:subagent:abc-123": {
    "sessionId": "uuid-v4-string",      // 会话的内部唯一 ID
    "createdAt": 1715000000000,         // 创建时间戳
    "updatedAt": 1715000060000,         // 最后活跃时间
    "deliveryContext": {                // 消息投递上下文 (路由信息)
      "channel": "telegram",
      "accountId": "bot_1",
      "to": "user_123",
      "threadId": null
    },
    "sessionFile": "/path/to/transcripts/uuid.jsonl" // 对话记录文件路径
  }
}
```

### 2.2 并发控制
为了防止多进程/多请求同时写入导致数据损坏，OpenClaw 实现了文件锁机制 (`acquireSessionWriteLock`) 和原子写入 (`writeTextAtomic`)。

### 2.3 缓存机制
系统会对 Session Store 进行内存缓存 (`store-cache.ts`)，并支持 TTL (Time-To-Live) 过期策略，以减少高频读取时的磁盘 I/O 开销。

## 3. 对话记录 (Transcripts)

实际的聊天记录（User 和 Assistant 的对话内容）并不存储在 `store.json` 中，而是独立存储为 **JSONL** (JSON Lines) 格式的 Transcript 文件。

### 3.1 存储路径
默认路径通常为 `sessions/<sessionId>.jsonl`。

### 3.2 记录格式
每一行是一个独立的 JSON 对象，代表一条消息或事件：
```json
{"type":"header","version":1,"id":"uuid","timestamp":"..."}
{"role":"user","content":[{"type":"text","text":"你好"}],"timestamp":...}
{"role":"assistant","content":[{"type":"text","text":"您好，我是丞相。"}],"timestamp":...}
```

### 3.3 读写操作 (`transcript.ts`)
- **写入**: 使用 `appendAssistantMessageToSessionTranscript` 追加新消息。
- **读取**: 系统通过流式读取 JSONL 文件来恢复上下文窗口 (Context Window)。

## 4. 维护与清理 (Maintenance)

为了防止磁盘空间耗尽，OpenClaw 内置了自动维护机制 (`store-maintenance.ts`)：

- **Pruning (修剪)**: 定期清理超过 `pruneAfterMs` (如 30 天) 未活跃的会话 Entry。
- **Capping (上限)**: 限制单个 Store 文件中的最大会话数 (`maxEntries`)，超出时按 LRU 策略移除。
- **Archiving (归档)**: 被移除的会话 Transcript 文件会被移动到归档目录或删除。

## 5. 总结

OpenClaw 的会话管理设计遵循 **"元数据与内容分离"** 的原则：
- `store.json` 轻量级，负责路由和索引。
- `transcript.jsonl` 负责持久化海量对话历史。
这种设计既保证了路由查询的高性能，又提供了可靠的对话持久化能力。
